# Лабораторная работа 2 (Azure vs AWS). Отчет

## Цель
Получение навыков аналитики и понимания спектра публичных облачных сервисов без привязки к вендору. Формирование единой кросс-провайдерной сервисной модели на основе подхода из ЛР1.

## Задача
Сопоставить входящие данные от провайдера Microsoft Azure с его документацией и разложить потребление по той же иерархии, что использовалась в ЛР1 (AWS).

Входные поля Azure, по которым выполнялось сопоставление:
- Consumed Service
- Meter Category
- Meter Sub-Category
- Meter Name

Заполняемые поля (иерархия сервисной модели):
- IT Tower
- Service Family
- Service Type
- Service Sub Type
- Service Usage Type

## Самые полезные источники информации
- https://azure.microsoft.com/en-us/products/
- https://azure.microsoft.com/en-us/products/category/
- https://learn.microsoft.com/en-us/azure/reliability/availability-service-by-category

## Как выполнялась работа

### 1. Импорт и подготовка таблицы
1) Если исходник был в CSV: импорт в Excel через Данные → Из текста/CSV, разделитель “;”.
2) Проверка, что строки корректно распарсились, а колонки соответствуют ожидаемым полям биллинга (Consumed Service / Meter Category / Meter Sub-Category / Meter Name).

### 3. Сопоставление строк биллинга с сервисами Azure
Для каждой строки из биллинга:
1) Смотрел Consumed Service и значения Meter Category / Meter Sub-Category / Meter Name.
2) По этим данным определял конкретный сервис Azure (например, Virtual Machines, Storage, SQL и т.п.).
3) Для уточнения использовал:
   - каталог продуктов Azure (для понимания назначения сервиса и названия продукта),
   - разбиение продуктов по категориям (для привязки к крупным категориям и согласованности),
   - список сервисов по категориям на Microsoft Learn (как дополнительная проверка классификации).

### 4. Привязка к общей кросс-провайдерной модели (как в ЛР1)
Правило: верхние уровни иерархии (IT Tower и Service Family) выбирались так, чтобы совпадать по смыслу с моделью из ЛР1.
То есть Azure-сервисы раскладывались в те же “ветки” (Compute, Storage, Networking, Databases, Cloud Services и т.д.), чтобы затем можно было сравнивать AWS и Azure в одной аналитике.

### 5. Заполнение 5 классификационных колонок
Правило заполнения:
- IT Tower и Service Family брались в рамках той же логической концепции, что и в ЛР1 (как в образце).
- Service Type и Service Sub Type фиксировал конкретный сервис Azure и его подсервис (по Meter Category / Meter Sub-Category).
- Service Usage Type фиксировал тип потребления по Meter Name, используя шаблоны с % там, где описание в биллинге допускает вариативность.
